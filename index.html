Conceito fundamental

O que é um SGBD?

SGBD significa Sistema de Gerenciamento de Banco de Dados. Ele é o software que armazena, organiza, consulta, protege e controla os dados.

  • Recebe comandos SQL.
  • Verifica permissões.
  • Controla integridade e relacionamentos.
  • Permite múltiplos acessos.
  • Organiza dados em bancos, tabelas, campos e registros.
  • Apoia recuperação em caso de falha.
Observação: O SGBD funciona como o centro de controle das informações.

Como o MySQL trabalha

Arquitetura cliente-servidor

No modelo cliente-servidor, uma ferramenta cliente envia comandos SQL para o servidor. O servidor processa e devolve o resultado.

  • Cliente: MySQL Workbench, sistema web, aplicativo, API ou terminal.
  • Servidor: MySQL Server, responsável por processar o comando.
  • Banco de dados: local lógico onde as tabelas ficam organizadas.
  • Conexão: caminho usado para comunicação entre cliente e servidor.
MySQL Workbench  →  MySQL Server  →  Banco de Dados
Cliente          →  Servidor      →  Dados

Fluxo de execução

O que acontece em um SELECT?

Quando executamos uma consulta, o comando passa por etapas até o resultado aparecer na tela.

  • O cliente envia o SQL ao servidor.
  • O servidor interpreta o comando.
  • O servidor verifica permissões.
  • O servidor localiza os dados solicitados.
  • O resultado retorna para o cliente.

Acesso simultâneo

SGBD multiusuário

Um SGBD multiusuário permite que várias pessoas acessem o banco ao mesmo tempo, sem perder a consistência dos dados.

  • Atendente cadastra cliente.
  • Estoquista atualiza produto.
  • Gerente consulta relatório.
  • Sistema registra venda.
  • Administrador executa manutenção.
Observação: Em ambiente real, quase nunca existe apenas uma pessoa usando o sistema.

Concorrência e consistência

Problemas em ambiente multiusuário

Quando várias pessoas acessam o mesmo dado, o SGBD precisa controlar conflitos.

ProblemaExplicação
ConcorrênciaDois usuários tentam alterar o mesmo registro.
InconsistênciaOs dados ficam incoerentes após alterações simultâneas.
BloqueioUm usuário aguarda o outro finalizar a operação.
Falha de transaçãoA operação começa, mas não é concluída.
Permissão indevidaUsuário acessa algo que não deveria.

InnoDB e controle

Como o MySQL ajuda?

O MySQL, usando InnoDB, reduz riscos em ambiente multiusuário.

  • Transações agrupam operações.
  • Bloqueios protegem registros durante alterações.
  • Chaves estrangeiras preservam integridade relacional.
  • Permissões controlam quem pode acessar e alterar dados.
  • COMMIT confirma e ROLLBACK desfaz alterações não confirmadas.
Observação: O objetivo é manter os dados corretos mesmo com vários acessos ao mesmo tempo.

Tabela ordens de serviço

A ordem de serviço é a entidade central do protótipo.

CREATE TABLE ordens_servico (
    id_os INT AUTO_INCREMENT PRIMARY KEY,
    id_cliente INT NOT NULL,
    id_equipamento INT,
    id_tecnico INT,
    data_abertura DATE DEFAULT (CURRENT_DATE),
    data_fechamento DATE,
    status_os VARCHAR(30) DEFAULT 'Aberta',
    descricao_problema VARCHAR(255) NOT NULL,
    observacao VARCHAR(255),
    FOREIGN KEY (id_cliente) REFERENCES clientes(id_cliente),
    FOREIGN KEY (id_equipamento) REFERENCES equipamentos(id_equipamento),
    FOREIGN KEY (id_tecnico) REFERENCES tecnicos(id_tecnico),
    CHECK (status_os IN ('Aberta', 'Em atendimento', 'Finalizada', 'Cancelada'))
) ENGINE=InnoDB;

    data_nascimento,

    os.data_fechamento

INNER JOIN clientes c ON os.id_cliente = c.id_cliente
LEFT JOIN equipamentos e ON os.id_equipamento = e.id_equipamento
LEFT JOIN tecnicos tec ON os.id_tecnico = tec.id_tecnico;

SELECT * FROM vw_resumo_ordens;

Continuidade

Backup e recuperação de falhas

Backup é uma cópia de segurança dos dados. Ele permite recuperar informações quando ocorre falha, erro ou exclusão acidental.

  • Exclusão acidental.
  • Erro em comando SQL.
  • Falha no servidor.
  • Corrupção da base.
  • Problema no computador.
  • Ataque ou acesso indevido.
  • Troca ou migração de sistema.
Observação: Banco sem backup está sempre em risco.
